# Test — Auth UI (Stage 1, 2 & 3)

Эта ветка содержит статический интерфейс (ЭТАП 1), UI-стили и клиентскую логику (ЭТАП 2) и интеграцию с backend (ЭТАП 3).

Что реализовано (ЭТАП 1)
- Статичный интерфейс авторизаци�� (карточка) с вкладками "Вход" и "Регистрация".

Что реализовано (ЭТАП 2)
- Финальная стилизация интерфейса (hover, disabled, responsive)
- Клиентская валидация всех полей
- Отображение ошибок рядом с полями
- UI-состояния: idle / loading / error / success
- Блокировка кнопок во время loading
- Очистка полей после success

Что реализовано (ЭТАП 3)
- Интеграция с endpoint'ами логина и регистрации (конфигурируются в `src/app.js` как `/api/login` и `/api/register`).
- Обработка успешных и ошибочных ответов сервера.
- Маппинг серверных ошибок (формат `errors: { field: message }`) в UI — сообщения показываются рядом с полями.
- Отображение общих ошибок сервера в статусе формы.
- UI-состояния: loading / error / success — без редиректов и без управления сессией.

Ограничения
- Не выполняются редиректы и не производится управление сессиями в этом коде.
- API-энпойнты ожидают JSON и могут возвращать структуру ошибок в поле `errors`.

Как протестировать с backend
1. Поднимите сервер и обеспечьте следующие endpoint'ы:
   - POST /api/login — ожидает JSON { email, password, ... }.
   - POST /api/register — ожидает JSON с полями регистрационной формы.
2. На ошибках сервер может возвращать HTTP 400/422 и JSON вида:
   {
     "errors": { "email": "Неверный email", "password": "Слабый пароль" }
   }
   или
   {
     "message": "Общая ошибка процесса"
   }
3. После корректного ответа сервер должен вернуть 200/201 и опционально { message: '...' }.
